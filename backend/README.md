# Backend

`backend/` отвечает за API, хранение чатов, потоковую генерацию ответов и обслуживание контекста для LLM.

## Зона ответственности

- принимать HTTP-запросы от frontend и валидировать их;
- хранить чаты и сообщения в PostgreSQL;
- стримить ответ модели через SSE;
- поддерживать длинные диалоги через суммаризацию старых сообщений;
- генерировать заголовок чата после первого сообщения.

## Почему выбрана такая архитектура

### FastAPI

FastAPI выбран по трем причинам:

- проект строится вокруг асинхронного I/O: БД, HTTP-запросов к LLM и стриминга ответа;
- коробочный OpenAPI и типизированные схемы ускоряют разработку и дебаг;
- зависимостями удобно собирать сервисы и репозитории без ручного DI-контейнера.

### Разделение на `api -> services -> repositories`

Такое разделение выбрано намеренно:

- `api/` отвечает только за HTTP-слой, статусы, валидацию и orchestration;
- `services/` содержит бизнес-логику: создание чатов, запись сообщений, суммаризацию, генерацию title;
- `repositories/` инкапсулируют SQLAlchemy-запросы и дают предсказуемую точку доступа к данным.

Это уменьшает связность и позволяет менять реализацию хранения без переписывания маршрутов.

### PostgreSQL

PostgreSQL здесь логичнее, чем NoSQL, потому что домен в основном реляционный:

- у чата есть упорядоченный список сообщений;
- важны транзакции при записи сообщения и обновлении счетчиков;
- нужны индексы, сортировка, пагинация и каскадные удаления.

### SQLAlchemy Async + Alembic

- async SQLAlchemy соответствует асинхронной модели FastAPI и не блокирует event loop на уровне драйвера;
- Alembic нужен для контролируемых миграций и воспроизводимого состояния схемы в любой среде.

### SSE, а не WebSocket

В этом приложении поток всегда однонаправленный: сервер отправляет токены ответа клиенту. Для такого сценария SSE проще и дешевле в сопровождении:

- нет отдельного протокола поверх HTTP;
- проще интеграция с proxy и dev-инфраструктурой;
- легче отлавливать ошибки и повторять запрос обычными HTTP-инструментами.

### Суммаризация как часть backend, а не отдельный сервис

Суммаризация находится в `ContextService`, потому что это часть правил домена, а не инфраструктурная мелочь:

- backend знает пороги, размер batch и правила сборки контекста;
- одно место отвечает за то, что попадет в модель;
- сохраняется детерминированное поведение без внешнего очередного слоя.

### Post-stream задачи вынесены отдельно

Файл `services/stream_tasks.py` существует не случайно. Он нужен, чтобы не задерживать выдачу токенов клиенту второстепенной работой:

- сначала клиент получает поток модели;
- потом выполняются title generation и summarization;
- latency первого ответа остается ниже.

## Основные модули

- `app/main.py` — создание приложения, lifespan, CORS, healthcheck.
- `app/api/v1/chats.py` — CRUD чатов.
- `app/api/v1/messages.py` — получение сообщений, отправка и регенерация через SSE.
- `app/services/chat_service.py` — операции над чатом и генерация названия.
- `app/services/message_service.py` — запись и обновление сообщений.
- `app/services/context_service.py` — сборка контекста и суммаризация.
- `app/services/llm_service.py` — клиент OpenRouter.
- `app/repositories/` — SQLAlchemy-слой доступа к данным.
- `alembic/` — миграции схемы.

## Почему конфигурация устроена через отдельные DB-поля

Приложение не использует один `DATABASE_URL` в `.env`, а собирает DSN из `POSTGRES_HOST`, `POSTGRES_PORT`, `POSTGRES_DB`, `POSTGRES_USER`, `POSTGRES_PASSWORD`. Это выбрано по практической причине:

- проще переключаться между `localhost` и Docker hostname `postgres`;
- `docker-compose.yml`, backend runtime и Alembic читают один и тот же набор переменных;
- меньше риска расхождения между runtime-подключением и миграциями.

## Локальный запуск

```bash
cd backend
cp .env.example .env
# Для локального запуска укажи POSTGRES_HOST=localhost
python3 -m pip install -r requirements.txt
python3 -m alembic upgrade head
python3 -m uvicorn app.main:app --reload
```

## Через Docker

Из корня репозитория:

```bash
make env-init
make docker-up-build
make docker-migrate
```

## Тесты

Локально:

```bash
make backend-test
```

## Текущие trade-offs

- аутентификации нет, вместо нее используется `X-Client-Id` для изоляции данных;
- нет отдельной очереди задач, поэтому post-stream работа выполняется в рамках процесса backend;
- суммаризация хранится как текст в чате, а не как отдельная knowledge-модель или vector store.

Это оправдано текущим размером задачи: проект остается простым, прозрачным и быстрым в разработке.
